# Universal Application Shell Post-Migration Audit

Date: 2026-06-30

## Executive Summary

The universal shell is installed globally through `layouts.app` and `layouts.admin`, so every authenticated app/admin Blade page now receives the shared shell wrapper and ordered shell-region contract. The production risk is low for the baseline wrapper: omitted regions render safely, existing content still loads, and existing smoke coverage passes.

The page-level shell-section migration is complete for the audited authenticated app/admin surfaces. P1, P2, and P3 rows now adopt real shell sections for page titles/descriptions, primary actions, filter bars where safe, and KPI sections via `x-metric-section`/`x-metric-card` where present. Dense table, drawer, confirmation-form, and workspace-specific interaction surfaces remain local where moving them would change behavior. Those local areas are intentionally deferred until they can be migrated without changing table, drawer, focus, form, or workflow semantics.

Priority definitions:

- `P1`: high-traffic or structurally dense pages with multiple shell-region candidates: headers, filters, KPI grids, action clusters, tables, drawers or card-heavy layouts.
- `P2`: moderate migration complexity or detail pages with repeated section/card patterns.
- `P3`: simple forms, low-density list pages, or pages needing only header/action extraction.

P1 migration notes:

- Migrated P1 rows are marked `Shell sections adopted`. Their historical `issues found` / `recommended fix` columns are retained as audit context for remaining non-shell cleanup rather than reset as completed checklist text.
- Deferred: dense local content filters on `resources/views/app/content/index.blade.php` remain in content until the table/filter workflow can be extracted without changing behavior.
- Deferred: existing drawer-like workspaces remain local unless they can be moved to `detailDrawer` without changing focus, overlay, or form behavior. Universal DataTable work is tracked separately in `docs/universal-data-table-migration.md`.

P2 migration notes:

- Migrated P2 rows are marked `Shell sections adopted`. Page titles/descriptions were moved to `pageHeader`; obvious top-level actions moved to `primaryActions`; safe GET filters and KPI grids moved to `filterBar`/`metricSection` where they were behavior-neutral.
- Deferred: dense review/workspace action clusters, especially comparison, opportunity, programmatic review, queue action, and LLM tracking detail controls, remain local because their confirmation forms, sticky context, focus order, or workflow semantics are tightly coupled to the content region.
- Deferred: table internals and dense table-adjacent filters remained unchanged during the shell pass unless they were simple top-level GET filters. Universal DataTable work is tracked separately in `docs/universal-data-table-migration.md`.

P3 migration notes:

- Migrated P3 rows are marked `Shell sections adopted`. All remaining baseline pages now provide a shell `pageHeader`; straightforward top-level actions or simple GET filters were moved where behavior-neutral.
- Deferred: dense forms, drawer-like billing and workspace areas, table internals, confirmation actions, and table-adjacent filters remain local where moving them would risk route, focus, form, or workflow semantics.
- Final shell adoption status: P1, P2, and P3 authenticated app/admin pages in this audit now adopt real shell sections. Universal DataTable migration is now a separate follow-on phase.

## Shell Contract Readiness

- Breadcrumb, page header, description, primary actions, filter bar, KPI section, detail drawer and footer actions render safely when omitted. Empty regions are present in the component template but hidden by CSS when they have no child elements.
- `x-filter-bar` remains backward compatible. Existing usages continue passing arbitrary slot content, while the component can now accept optional title/description/actions.
- App/admin layout behavior is preserved. Navigation, support-mode banners, search, notification menus, language selection, impersonation and page-width handling remain outside the new page-region contract.
- The shell introduces a structural wrapper around existing page content. It is safe, but some dense pages still retain local `mb-*`, `space-y-*`, `rounded-lg border bg-surface`, or custom workflow sections until those areas can be simplified without changing behavior.
- No production behavior fixes were made in this audit.

## Visual And UX Findings

- The shared shell wrapper aligns app/admin pages at the layout level, but visual alignment remains inconsistent because page content still owns its own headers, cards, filters and tables.
- The strongest duplicated patterns are:
  - Local `<h1>` plus description blocks at the top of content.
  - Local action bars built with `inline-flex`, `pl-btn-*`, or raw buttons.
  - Local GET filter forms, sometimes inside `x-filter-bar`, sometimes inside ad hoc bordered cards.
  - KPI grids built from repeated `rounded-lg border border-border bg-surface` cards.
  - Detail pages with repeated nested cards instead of `x-section-container`.
  - Existing local header components, especially `x-app.insights-header` and `x-app.content-area-header`, which now duplicate the shell's page header/action responsibilities.
- Mobile behavior should remain functionally safe because the shell wrapper does not introduce fixed-width constraints. The next migration phase should still check dense P1 pages manually because several local grids and card clusters predate the shell rhythm.
- Dark mode compatibility is not complete. The app uses light theme tokens and semantic colors but no audited dark-mode contract. The shell does not regress this, but dark mode should not be claimed production-ready yet.
- Empty states are inconsistent. Some pages use local dashed cards, some use prose-only empty rows, and some use domain-specific empty state components. Future page migration should consolidate app/admin empty states on `x-empty-state`.

## Accessibility Findings

- Landmarks: app/admin layouts still expose a single `<main>` region. The shell adds a `data-shell-region="main-content"` wrapper but no extra landmark, so landmark structure remains acceptable.
- Heading order: most pages still render local `<h1>` inside content. This is semantically valid today, but after moving titles into `pageHeader`, those local `<h1>` elements must be removed or demoted to avoid duplicate page-level headings.
- Focus states: existing global focus-visible styles still apply. New shell primitive buttons/links inherit the same tokenized focus rings.
- Drawer compatibility: `x-drawer-container` exists, but pages have not adopted it yet. Before production drawer usage, add page-level focus management, close affordance, `aria-modal`/dialog semantics when modal, and keyboard escape behavior where appropriate.
- Action labels: many icon-only or compact actions are still local and should be reviewed during migration. Shell migration is a good point to ensure icon-only controls have `aria-label` or visible text.

## Test Coverage Gaps

- Current shell tests verify component rendering and that app/admin layouts include the shell contract.
- Existing admin/content smoke tests verify representative pages still load.
- Added coverage now verifies one migrated app P1 page and one migrated admin P1 page use `pageHeader`, `primaryActions`, `filterBar`, `metricSection`, and metric components.
- Missing coverage:
  - A regression test that old `x-filter-bar` usages render with and without optional title/actions.
  - Browser or snapshot coverage for migrated dense P1 pages.
  - Accessibility-focused tests for drawer open/close behavior once drawers are actually wired to `detailDrawer`.
  - Mobile viewport checks for dense migrated pages, especially content index, dashboard, LLM tracking and admin queues.

## Recommended Migration Order

1. Migrate P1 pages with dense operational workflows first: app dashboard, content index/show/calendar, drafts show, LLM tracking index, admin dashboard, admin queues, admin LLM monitor and admin users.
2. Refactor local header components into shell-section producers rather than maintaining them as parallel header systems.
3. Move GET filter forms into `filterBar` before touching table internals.
4. Move KPI grids into `metricSection`/`metricCard` before the DataTable phase.
5. Standardize `sectionContainer`, `sectionHeader`, `emptyState` and `errorState` usage across detail pages.
6. Keep the DataTable phase separate; tables should only be wrapped and visually normalized in this phase.
